## Telegram Libraries

There are three distinct tiers depending on how deep you want to go.

### Bot API tier (HTTP-based)

**python-telegram-bot** (PTB) is the gold standard for pure bot development. It provides a fully async Python interface for the Telegram Bot API and natively supports all types and methods of Bot API 9.5. Beyond the raw API, it includes convenience shortcuts and high-level classes in `telegram.ext` that make bot development straightforward. The latest stable release is v22.6 (January 2026), requiring Python 3.10+.

**aiogram** is the other major async-first option. It's a modern, fully asynchronous framework for the Telegram Bot API built on asyncio and aiohttp, designed for low-ceremony bot setup.

**pyTelegramBotAPI** (telebot) is a more accessible choice for those new to async. It supports both synchronous and asynchronous patterns and is tested against Python 3.9–3.13. It uses a `TeleBot` class that wraps all API calls and handles middleware.

### MTProto tier (direct protocol access)

**Telethon** is in a different class. Rather than going through the HTTP Bot API, Telethon communicates directly with Telegram's servers via the MTProto protocol — the same one used by official Telegram clients. This means there's no HTTP connection, no polling, no webhooks, and no JSON overhead. This is critical for userbots — automating actions as a regular user account — monitoring channels without joining, bulk media downloads, and anything not accessible through the standard Bot API. The latest release is Telethon 1.42.0 with full support for Telegram API layer 201.

**Pyrogram** is Telethon's direct competitor in the MTProto space, often praised for a more Pythonic, object-oriented feel. It's worth checking if you prefer OOP-style abstractions.

**When to pick which**: If you need to interact with Telegram as a user account, monitor channels, or work with advanced MTProto features, Telethon is the superior choice. python-telegram-bot excels for simple bot API implementations.

## Discord Libraries

The ecosystem here fragmented when discord.py briefly went dormant, spawning a cluster of maintained forks.

**discord.py** (Rapptz) is back and actively maintained. The current version is 2.7.1 (March 2026), with a modern async/await API, proper rate limit handling, and optimized memory usage.

**disnake** is a fork of discord.py with strong slash-command support. Latest stable is 2.12.0 (February 2026), requiring Python 3.10+. It's fully typed and production-stable. Disnake revolves around an event-driven architecture — you register listeners for events like `on_message`, `on_connect`, `on_ready`, `on_shard_connect`, and many others. A class-based events system (new in v2.8) offers autocompletion and documentation support.

**hikari** takes a different approach. It's a statically typed Discord microframework supporting Discord's v10 REST and Gateway APIs, supporting Python 3.10–3.14. It provides two implementations: `GatewayBot` (receives all events via WebSocket) and `RESTBot` (interaction-server-only, for slash commands via webhook). The tradeoff is that hikari is more explicit and lower-level, which is valuable for performance-critical or large bots but steeper for beginners. You'd pair it with **lightbulb** or **tanjun** for command handling.

**interactions.py** rounds out the major options as a more interaction-focused framework built from the ground up.

## OpenRouter

OpenRouter provides a unified API giving access to hundreds of AI models through a single endpoint, automatically handling fallbacks and routing. It's OpenAI-compatible, meaning you can often just swap the `base_url` in existing OpenAI SDK code.

**Official Python SDK**: The OpenRouter Python SDK is type-safe, built on Pydantic, auto-generated from OpenRouter's OpenAPI spec, and gives access to 300+ models through a unified interface. Invalid configs are caught at runtime with clear error messages.

**Event/streaming model**: OpenRouter uses Server-Sent Events (SSE) for streaming. It occasionally injects comment payloads to prevent connection timeouts — these should be ignored per SSE spec. Cancellation of streaming requests is supported for some providers, which stops model processing and billing immediately. The SDK has streaming built in, with custom event transforms, tool integration, and a modular agent-oriented design. TypeScript and Python SDKs are both available.

Async streaming usage looks like:
```python
async with OpenRouter(api_key=os.getenv("OPENROUTER_API_KEY")) as client:
    async for event in await client.chat.send_async(
        messages=[{"role": "user", "content": "..."}],
        model="anthropic/claude-sonnet-4-5",
        stream=True
    ):
        print(event)
```

Streaming just requires adding `"stream": true` to the request and handling SSE on the response. Provider routing can also be tuned — e.g. `provider: {"sort": "price"}` for cost optimization or `{"sort": "latency"}` for speed.
